# System Scope — Tabletop RPG Platform

## Vision

An all-in-one platform for Game Masters to build, manage, and run tabletop RPG campaigns. Covers the full lifecycle: world-building (wiki), character creation, map design, and live game sessions — all connected by a context-aware AI assistant.

---

## Bounded Contexts

The bounded context split is the most important architectural decision. Each BC has exclusive data ownership, an independent lifecycle, and a distinct ubiquitous language.

The central rule for authorization: **Campaigns manages membership and coarse-grained roles (GM vs Player). Each BC manages its own fine-grained, resource-level access rules** using the identity and role provided by IAM and Campaigns.

---

### 1. IAM (Identity & Access Management)

Answers one question: *who are you, and are you authenticated?*

- User registration, login, password management
- JWT issuance, refresh tokens, session invalidation
- User profile (display name, avatar)
- Does NOT own campaign roles or resource permissions — those live in the BC that owns the resource

---

### 2. Campaigns

The top-level organizing concept of the entire platform. Everything else is scoped to a campaign.

- Campaign creation, naming, settings, archival
- Participant management: invite by email, accept/reject invitations
- Coarse-grained roles: **Game Master** (owner, full control) and **Player** (invited participant, scoped access)
- A user can be GM of one campaign and Player in another — roles are campaign-scoped

---

### 3. Wiki

World-building knowledge base. The primary creative tool for GMs.

**Features:**
- Rich text documents (headings, tables, embeds, inline images)
- Folder hierarchy within a campaign
- Read mode (clean rendered view)
- **Resource-level sharing**: share specific folders or documents with specific players (Viewer or Contributor), overriding campaign defaults
- In-document image generation via AI (the Assistant BC handles the AI call; the result becomes an Asset embedded here)
- "Show to Players" handout button — pushes a document into the live Session for players to view

---

### 4. Compendium

The game rules reference library. A *Generic Supporting Domain* — read-only, system-authored data.

**Features:**
- Stores D&D 5e 2014 SRD content: classes (with features per level), races, backgrounds, spells, feats, abilities, monster templates
- Structured to support future game systems without changes to other BCs
- Read-heavy; rarely changes (only on errata or new system additions)
- Consumed by Characters (to build and level up PCs) and Session (for combat resolution rules)

---

### 5. Characters

Manages the instantiation of game rules into playable characters and monster stat blocks.

**Features:**
- Create and edit **player character sheets** (using Compendium data for D&D 5e)
- Create and edit **monster sheets** (from Compendium templates or custom)
- **Level-up workflow** — guided flow applying choices from Compendium (ASI, new class features, spells, etc.); accessible from the Characters page AND from within the character sheet modal during a live session
- Tracks runtime character state: current HP, used spell slots, active conditions
- Exposes a public interface so Session can read/update state during live play
- A character belongs to a campaign and optionally to a player (owner)

---

### 6. Assets

The campaign media library. A *Supporting Domain* — no unique business logic, but shared state.

**Features:**
- Upload images (map backgrounds, token artwork, portraits, handout illustrations)
- Import from external URLs
- Tag, name, and organize assets within a campaign
- Referenced by: Wiki (embedded images), Maps (backgrounds + token images), Characters (portraits), and images generated by the Assistant

---

### 7. Maps

Design-time map authoring. Produces map definitions that Sessions reference at runtime.

**Features:**
- Create maps from scratch or from an imported background image (via Assets)
- **Layers:**
  - **Map layer** — terrain, objects, background elements
  - **Tokens layer** — player/monster/NPC tokens with optional Character link
  - **GM layer** — markers and notes invisible to players
- Token placement and association with Characters/Monsters
- **Drawing and measuring tools** available in edit mode (same tools as in Session, but non-real-time)
- A map is a *template/definition* — live state (positions, fog of war, drawings, measurements) is owned by Session

---

### 8. Session (Live Gaming)

Real-time game orchestration. The most infrastructure-intensive BC (WebSocket-heavy, event-driven).

**Features:**
- Start a session from a campaign map; share live view with invited players
- Real-time token movement (GM and players with permission, via WebSocket)
- **Fog of war** — GM controls player visibility on the map
- **Drawing tools** over the map — GM and players can draw; drawings are broadcast in real-time and visible to all participants
- **Measuring tools** — rulers and directional arrows for distance measurement; both GM and players can use them during edit mode or in a live session; measurements are visible to all participants in real-time
- **GM map editing during session** — dynamic layer editing without stopping play
- **Combat system:**
  - Initiative tracker
  - Turn management (HP changes, conditions, actions)
  - Automated rolls sourced from character sheet stats
- **Handouts** — GM pushes Wiki pages to players as in-session modals; detachable to separate browser window
- **Character/Monster sheet modals** — viewable in-session, detachable
- Reads map definition from Maps BC; owns all live runtime state itself

---

### 9. Assistant

Context-aware AI companion available across the entire platform.

**Features:**
- Persistent conversation interface, context-aware based on current location:
  - **Wiki** → lore writing help, NPC descriptions, world-building suggestions, image generation
  - **Characters** → stat block help, spell lookups, level-up guidance
  - **Maps** → scene descriptions, encounter design
  - **Session** → rules lookups, combat helper, monster tactics
- **Image generation** — prompt → AI model API → result stored as Asset → reference returned to caller
- Iterative image refinement (follow-up prompts, re-generation)
- Tool registry: the Assistant calls the same API endpoints as the frontend (dual-purpose API principle)

---

## Cross-Cutting Concerns

| Concern | Responsibility |
|---|---|
| **Authentication** | IAM (JWT middleware consumed by all BCs) |
| **Coarse authorization** | Campaigns (GM vs Player role, resolved per campaign) |
| **Fine-grained authorization** | Each BC (e.g., Wiki enforces per-folder ACL, Session enforces token ownership) |
| **Real-Time** | Session BC owns WebSocket infrastructure; other BCs are request/response |
| **File Storage** | Assets BC owns S3-compatible object storage interface |
| **Internal Events** | Domain events between BCs (e.g., Character state updated → Session notified) |
| **Observability** | Structured logging + health checks per BC; shared infra config |

---

## Out of Scope (v1)

- Voice/video chat in sessions
- Mobile apps — **not in v1, but the architecture must support it**: API-first design, no server-side rendering assumptions, responsive-ready contracts. Expected mobile scope: character management only (not live gaming), though this may expand later
- Community content sharing / marketplace
- Offline mode

## Future-Proofing Constraints

These are not v1 features but must not be violated by v1 architecture decisions:

- **Multiple game systems** — Compendium is system-agnostic by design; UI only exposes D&D 5e 2014 initially
- **Mobile apps** — all APIs must be stateless and mobile-consumable (no session cookies, JWT only); initial target is character management; frontend components should be layout-agnostic where possible
- **Extracting BCs to microservices** — module boundaries must be strict enough that Session or Characters could be deployed independently if scale demands it

## Long-Horizon Ideas (No Timeline)

Worth capturing so they don't constrain architecture decisions but also aren't forgotten:

- **3D dice animations** — physics-based dice rolls visible to all participants at the table in real-time (requires a broadcast channel, likely the same WebSocket infrastructure as Session)
- **Spell/magic animations** — visual effects users can trigger during a session (fireball, lightning bolt, etc.), broadcast to all participants
- **Mobile live gaming** — currently out of scope for mobile, but don't actively close the door
